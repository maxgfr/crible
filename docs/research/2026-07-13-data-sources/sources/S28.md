# EDINET for Developers: The Complete English Guide
Onkar Dahale
There is no English-language developer guide to EDINET. The official documentation is a 97-page Japanese PDF. Stack Overflow has maybe three questions about it. The best resources are scattered Qiita posts that assume you already know what a 有価証券報告書 is.
We built Axiora on top of EDINET, so we wrote the guide we wish existed when we started. This is everything a developer needs to know to work with Japanese financial data — whether you use our API or go straight to the source.
## What EDINET Is (and Isn't)
EDINET (Electronic Disclosure for Investors' NETwork) is Japan's statutory filing system for corporate securities disclosures. Think of it as Japan's SEC EDGAR — except the API is date-based, the docs are Japanese-only, and the XBRL taxonomy has three accounting standards instead of one.
It's run by the Financial Services Agency (金融庁 / FSA), which is Japan's equivalent of the SEC. Every listed company in Japan — plus investment funds, REITs, and any entity that publicly offers securities — must file through EDINET. That's 4,125+ listed companies, all with machine-readable financial data going back roughly a decade.
The data itself is excellent. The developer experience of getting it out is not.
## The Ecosystem: EDINET, TDnet, J-Quants
Japan doesn't have one financial data system. It has three, and they each do different things.
EDINET has the full statutory filings — annual reports (有価証券報告書 / "yuho"), semi-annual reports, registration statements. These are the comprehensive, audited documents with complete financial statements in XBRL. The API is free. The filings show up weeks or months after the fiscal year ends.
TDnet has the real-time stuff — earnings announcements, dividend decisions, M&A notifications. When Toyota reports earnings, TDnet gets it the same afternoon. EDINET gets the detailed statutory filing a month later. TDnet's API costs ¥70,000/month.
J-Quants is JPX's developer-friendly service — stock prices and processed financial data via clean JSON. Free tier with a 12-week data delay; paid plans for real-time. Good for market data, but the underlying financial detail still comes from EDINET filings.
We ingest from EDINET — the primary source of truth for Japanese corporate financials.
## What's in a Filing
The Annual Securities Report (yuho) is the flagship document. It's 100-200+ pages and includes everything:
Business overview — segments, subsidiaries, employees
MD&A — management's own narrative on the year
Risk factors — everything the company thinks could go wrong
Financial statements — the full set: balance sheet, income statement, cash flow, changes in equity — all XBRL-tagged
Corporate governance — board composition, executive compensation
Audit report — the independent auditor's signoff
The key fact for developers: every number in the financial statements is encoded in XBRL — an XML standard that tags each value with what it represents, what period it covers, and what currency it's in. This is what makes EDINET data machine-readable. It's also what makes it painful to extract.
## The EDINET API
EDINET v2 (launched April 2024) provides free programmatic access to all filings. You get an API key by creating a free account on the EDINET portal — no credit card, no approval process.
Base URL : https://api.edinet-fsa.go.jp/api/v2
curl "https://api.edinet-fsa.go.jp/api/v2/documents.json?date=2026-06-25&type=2" \
-H "Ocp-Apim-Subscription-Key: YOUR_KEY"
This returns every document filed on June 25th. A typical day during peak filing season returns 100-200+ results:
{
"metadata" : {
"resultset" : { "count" : 142 },
"status" : "200"
},
"results" : [
{
"docID" : "S100ABCD" ,
"edinetCode" : "E02144" ,
"secCode" : "72030" ,
"filerName" : "トヨタ自動車株式会社" ,
"docTypeCode" : "120" ,
"periodStart" : "2024-04-01" ,
"periodEnd" : "2025-03-31" ,
"xbrlFlag" : "1"
}
]
}
To download the actual filing: GET /documents/S100ABCD?type=1 returns a ZIP containing the XBRL files.
### The Problem Nobody Warns You About
The EDINET API is date-based only . There is no company search. There is no GET /companies/7203/filings endpoint. There is no way to ask "give me all filings by Toyota."
To find Toyota's annual report, you need to:
Know that Toyota's fiscal year ends in March, so the yuho lands sometime in June
Query every day in June: ?date=2026-06-01 , ?date=2026-06-02 , ?date=2026-06-03 …
Scan each day's 100+ results for edinetCode: "E02144"
Download the matching ZIP
To build a complete database, you iterate over every calendar date and process every filing. For a historical backfill — say, 5 years — that's ~1,800 daily API calls before you've parsed a single byte of XBRL.
We do this automatically, polling every 30 minutes. But if you're going direct, budget for it.
## Identifiers: Three Systems, No Translation API
Japan uses three overlapping identifiers for companies. None of them are interchangeable without a lookup table.
Identifier Toyota Who Issues It
EDINET Code E02144 FSA — identifies the filing entity
Securities Code 7203 JPX — the ticker you see on screens
ISIN JP3633400001 JASDEC — international standard
One gotcha: EDINET's secCode field returns 5 digits ( 72030 ). The trailing digit is a check digit. The actual ticker is 7203 .
There is no API to map between these. You download the EDINET Code List CSV and build the mapping yourself. We maintain this internally and expose both EDINET codes and securities codes in our API.
## Why XBRL Parsing Will Ruin Your Weekend
This is the section we wish someone had written for us before we started.
XBRL tags each financial value with a machine-readable element name — a namespace prefix plus a concept name. Find the element, grab the text, done. Right?
No. The element name for "revenue" depends on which accounting standard the company uses, which industry it's in, and which section of the report you're reading. There is no single revenue tag. JP-GAAP manufacturers, JP-GAAP service companies, banks, insurance companies, US-GAAP filers, and IFRS filers all use different element names for what is conceptually the same number.
That's 6+ elements for one field. Across all financial metrics, we map hundreds of XBRL elements to 52 normalized fields. The core fields (revenue, net income, total assets, EPS, cash flows) are fully populated; newer fields (gross profit, SGA, inventories, capex, R&D, and others) are currently backfilling across historical filings.
### Context: The Silent Bug Factory
Every XBRL value has a contextRef that encodes the reporting period and consolidation status. This is where most parsing bugs live.
A typical filing contains both consolidated and non-consolidated (parent-only) financials. The context IDs look like:
CurrentYearDuration — current year, consolidated, flow item
CurrentYearInstant — balance sheet date, consolidated
CurrentYearInstant_NonConsolidatedMember — parent-only
Parse the wrong context and you get the parent company's revenue instead of the group's. For Japan's largest companies, the difference between parent-only and consolidated group can be tens of trillions of yen. Getting context resolution right requires a priority system that considers period, consolidation scope, and accounting standard simultaneously.
### Two Formats, Because Why Not
Older filings use traditional XBRL — pure XML, sometimes 10-50MB. Modern filings use Inline XBRL (iXBRL) — HTML with embedded XBRL tags.
iXBRL has a subtle trap: scaling attributes. The displayed text in the HTML might be 29,929,992 , but a scale="6" attribute means you multiply by 10^6 to get the actual value: ¥29,929,992,000,000. Miss that attribute and every number in your database is off by a factor of a million.
~60% of recent filings use iXBRL. Your parser needs to handle both formats.
